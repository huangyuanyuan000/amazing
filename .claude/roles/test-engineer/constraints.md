# 测试工程师约束规则

## 测试覆盖率约束

### 单元测试覆盖率
- **最低要求**: 80%
- **推荐目标**: 90%
- **核心模块**: 必须达到 95%
- **检查时机**: 每次 PR 提交时自动检查
- **不达标处理**: PR 不能合并，必须补充测试用例

### 集成测试覆盖率
- **最低要求**: 60%
- **推荐目标**: 75%
- **关键接口**: 必须 100% 覆盖
- **检查时机**: 每次发布前检查
- **不达标处理**: 不能发布，必须补充集成测试

### E2E 测试覆盖
- **关键业务路径**: 必须 100% 覆盖
- **次要业务路径**: 至少 50% 覆盖
- **异常场景**: 至少覆盖常见异常
- **检查时机**: 每次发布前执行
- **不达标处理**: 评估风险后决定是否发布

### API 测试覆盖
- **所有 API 接口**: 必须 100% 覆盖
- **正常场景**: 必须覆盖
- **异常场景**: 至少覆盖 80%
- **边界条件**: 至少覆盖 70%
- **检查时机**: API 开发完成后立即测试

## 测试质量约束

### 测试用例质量
- **必须包含**:
  - 用例标题（清晰描述测试目的）
  - 前置条件（测试环境和数据准备）
  - 测试步骤（详细的操作步骤）
  - 预期结果（明确的验收标准）
  - 实际结果（执行后填写）
  - 测试数据（具体的测试数据）

- **禁止出现**:
  - 模糊的描述（如"测试功能正常"）
  - 缺少验收标准
  - 缺少测试步骤
  - 缺少测试数据

- **评审要求**:
  - 所有测试用例必须经过评审
  - 评审通过率 > 90%
  - 评审意见必须及时修改

### 自动化测试质量
- **稳定性要求**:
  - 自动化测试通过率 > 95%
  - 单个用例执行时间 < 30 秒
  - 整体测试套件执行时间 < 30 分钟
  - 不能有 flaky tests（不稳定的测试）

- **可维护性要求**:
  - 使用 Page Object 模式
  - 测试代码必须有注释
  - 测试数据与测试代码分离
  - 遵循 DRY 原则（Don't Repeat Yourself）

- **可读性要求**:
  - 测试用例命名清晰
  - 测试步骤逻辑清晰
  - 断言信息明确
  - 错误信息易于理解

## Bug 报告约束

### Bug 报告必须包含
1. **Bug 标题**: 简洁明确，描述问题现象
2. **Bug 描述**: 详细描述问题
3. **复现步骤**: 详细的操作步骤（必须可复现）
4. **预期结果**: 正确的行为应该是什么
5. **实际结果**: 实际发生了什么
6. **环境信息**: 操作系统、浏览器、版本等
7. **附件**: 截图、日志、视频等
8. **优先级**: Critical / High / Medium / Low
9. **严重程度**: Blocker / Major / Minor / Trivial

### Bug 优先级定义
- **Critical**: 系统崩溃、数据丢失、安全漏洞
- **High**: 核心功能不可用、严重影响用户体验
- **Medium**: 功能部分不可用、影响用户体验
- **Low**: 小问题、UI 问题、优化建议

### Bug 处理时效
- **Critical**: 2 小时内响应，24 小时内修复
- **High**: 4 小时内响应，3 天内修复
- **Medium**: 1 天内响应，1 周内修复
- **Low**: 3 天内响应，2 周内修复或延期

### Bug 验证要求
- **修复后必须验证**: 所有 Bug 修复后必须验证
- **验证通过标准**: 按照复现步骤验证，问题不再出现
- **回归测试**: 验证修复后不影响其他功能
- **验证不通过**: 重新打开 Bug，要求继续修复

## 时间约束

### 测试设计时间
- **简单功能**: 0.5 - 1 天
- **中等功能**: 1 - 3 天
- **复杂功能**: 3 - 5 天
- **超时处理**: 及时沟通，申请延期

### 测试执行时间
- **单元测试**: 每次 PR 提交时自动执行（< 5 分钟）
- **集成测试**: 每天自动执行（< 30 分钟）
- **E2E 测试**: 每次发布前执行（< 1 小时）
- **性能测试**: 每周执行一次（< 2 小时）
- **回归测试**: 每次发布前执行（< 2 小时）

### Bug 验证时间
- **Critical**: 修复后 2 小时内验证
- **High**: 修复后 4 小时内验证
- **Medium**: 修复后 1 天内验证
- **Low**: 修复后 3 天内验证

### 测试报告时间
- **日报**: 每天下班前提交
- **周报**: 每周五提交
- **发布报告**: 发布后 1 天内提交
- **专项报告**: 测试完成后 2 天内提交

## 质量标准约束

### 发布质量标准
- **Critical Bug**: 0 个
- **High Bug**: 0 个
- **Medium Bug**: < 5 个（评估后可接受）
- **Low Bug**: < 20 个（可延期修复）
- **测试覆盖率**: 达到要求
- **自动化测试**: 全部通过

### 不能放过的 Bug
- **Critical 级别**: 绝对不能放过
- **High 级别**: 绝对不能放过
- **安全漏洞**: 绝对不能放过
- **数据丢失**: 绝对不能放过
- **核心功能不可用**: 绝对不能放过

### 可以延期的 Bug
- **Low 级别**: 可以延期到下个版本
- **UI 优化**: 可以延期
- **性能优化**: 评估后可以延期
- **兼容性问题**: 评估后可以延期（非主流环境）

## 测试环境约束

### 测试环境要求
- **独立性**: 测试环境必须独立于开发环境和生产环境
- **稳定性**: 测试环境必须稳定可用
- **数据隔离**: 测试数据不能影响生产数据
- **环境一致性**: 测试环境尽量接近生产环境

### 测试数据要求
- **真实性**: 测试数据必须真实有效
- **完整性**: 测试数据必须覆盖各种场景
- **隔离性**: 测试数据不能影响其他测试
- **可重复性**: 测试数据可以重复使用

### 生产环境测试
- **禁止直接测试**: 不能直接在生产环境测试
- **需要审批**: 生产环境测试需要运维工程师审批
- **只读操作**: 只能执行只读操作
- **监控告警**: 必须有监控和告警

## 代码规范约束

### 测试代码规范
- **命名规范**:
  - 测试文件: `test_*.py` 或 `*_test.py`
  - 测试类: `Test*` 或 `*Test`
  - 测试方法: `test_*`
  - 清晰描述测试目的

- **结构规范**:
  - 使用 AAA 模式（Arrange-Act-Assert）
  - 一个测试只测试一个功能点
  - 测试之间相互独立
  - 避免测试之间的依赖

- **注释规范**:
  - 复杂测试必须有注释
  - 测试数据必须有说明
  - 特殊处理必须有注释

### 测试代码质量
- **代码审查**: 测试代码必须经过代码审查
- **代码覆盖**: 测试代码本身不需要测试
- **代码重构**: 定期重构测试代码
- **代码维护**: 及时更新过时的测试代码
