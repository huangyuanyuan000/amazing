# 固定角色说明

## 概述

固定角色是 Amazing 架构中的**基本分工**，存在于任何软件团队中。

这 6 种角色是软件开发的标准配置，不会因产品不同而改变。

---

## 为什么是这 6 种角色？

### 软件开发的基本分工

任何软件项目都需要：
1. **需求管理** → 产品经理（PM）
2. **技术决策** → 架构师（Architect）
3. **前端开发** → 前端开发（Frontend）
4. **后端开发** → 后端开发（Backend）
5. **质量保证** → 测试工程师（QA）
6. **部署运维** → 运维工程师（Ops）

这是经过实践验证的最小分工模型。

---

## 6 种固定角色

### 1. 👑 架构师 (Architect)

**职责**：
- 技术决策和架构设计
- 技术方案选型
- 代码审查和质量把关
- AI 模式切换
- 产品形态审查

**权限**：
- ✅ 查看所有代码
- ✅ 编写代码
- ✅ 审查代码
- ✅ 部署应用
- ✅ 切换 AI 模式
- ✅ 批准上线

**为什么固定**：
- 任何项目都需要技术决策者
- 架构设计是项目成功的关键
- 质量把关需要统一标准

**在 Sub-Agent 中的角色**：
- 参与架构设计
- 审查代码质量
- 把关技术方案

### 2. 📋 产品经理 (PM)

**职责**：
- 需求分析和管理
- PRD 编写
- 原型设计
- 功能验收

**权限**：
- ✅ 查看代码
- ✅ 创建需求
- ✅ 编写 PRD
- ✅ 验收功能
- ❌ 编写代码
- ❌ 部署应用

**为什么固定**：
- 任何项目都需要需求管理
- 产品规划需要专人负责
- 用户需求需要转化为技术需求

**在 Sub-Agent 中的角色**：
- 分析业务需求
- 编写功能 PRD
- 设计交互流程
- 验收功能实现

### 3. 🎨 前端开发 (Frontend)

**职责**：
- UI/UX 实现
- 组件开发
- 状态管理
- 前端性能优化

**权限**：
- ✅ 查看代码
- ✅ 编写前端代码
- ✅ 提交代码
- ❌ 审查代码
- ❌ 部署应用

**为什么固定**：
- 现代应用都需要前端界面
- 前端是独立的技术领域
- 需要专业的前端开发人员

**在 Sub-Agent 中的角色**：
- 实现 UI 界面
- 开发前端组件
- 对接后端 API
- 优化用户体验

### 4. ⚙️ 后端开发 (Backend)

**职责**：
- API 开发
- 业务逻辑实现
- 数据库设计
- 后端性能优化

**权限**：
- ✅ 查看代码
- ✅ 编写后端代码
- ✅ 提交代码
- ❌ 审查代码
- ❌ 部署应用

**为什么固定**：
- 任何应用都需要后端服务
- 后端是独立的技术领域
- 需要专业的后端开发人员

**在 Sub-Agent 中的角色**：
- 实现 API 接口
- 编写业务逻辑
- 设计数据库
- 优化性能

### 5. 🧪 测试工程师 (QA)

**职责**：
- 测试用例设计
- 自动化测试
- Bug 追踪
- 质量报告

**权限**：
- ✅ 查看代码
- ✅ 编写测试代码
- ✅ 提交 Bug
- ✅ 审查测试覆盖率
- ❌ 修改业务代码
- ❌ 部署应用

**为什么固定**：
- 质量保证是必需的
- 测试是独立的职能
- 需要专业的测试人员

**在 Sub-Agent 中的角色**：
- 设计测试用例
- 编写自动化测试
- 执行测试
- 报告 Bug

### 6. 🚀 运维工程师 (Ops)

**职责**：
- 应用部署
- 监控告警
- 故障排查
- 性能调优

**权限**：
- ✅ 查看代码
- ✅ 部署应用
- ✅ 配置监控
- ✅ 查看日志
- ❌ 修改业务代码
- ❌ 审查代码

**为什么固定**：
- 应用需要部署和维护
- 运维是独立的职能
- 需要专业的运维人员

**在 Sub-Agent 中的角色**：
- 部署应用
- 配置监控
- 处理告警
- 优化性能

## 角色协作

### 功能开发流程

```
PM（需求分析）
    ↓
架构师（技术决策）
    ↓
Frontend + Backend（并行开发）
    ↓
架构师（代码审查）
    ↓
QA（测试）
    ↓
架构师（验收）
    ↓
Ops（部署）
```

### Bug 修复流程

```
QA（发现 Bug）
    ↓
Backend（修复 Bug）
    ↓
架构师（审查修复）
    ↓
QA（回归测试）
    ↓
Ops（热修复部署）
```

| 角色 | 查看代码 | 编写代码 | 审查代码 | 部署 | 切换模式 | 批准上线 |
|------|---------|---------|---------|------|---------|---------|
| 架构师 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| PM | ✅ | ❌ | ❌ | ❌ | ❌ | ❌ |
| Frontend | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |
| Backend | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |
| QA | ✅ | ✅* | ✅* | ❌ | ❌ | ❌ |
| Ops | ✅ | ❌ | ❌ | ✅ | ❌ | ❌ |

*注：QA 可以编写测试代码，可以审查测试覆盖率

## 角色选择

### 使用 CLI 选择

```bash
# 选择角色
amazing-cli role select

# 可选角色：
# 0. 👑 架构师 (architect)
# 1. 📋 产品经理 (pm)
# 2. 🎨 前端开发 (frontend)
# 3. ⚙️ 后端开发 (backend)
# 4. 🧪 测试工程师 (qa)
# 5. 🚀 运维工程师 (ops)

# 直接设置角色
amazing-cli role set architect
```

### 查看当前角色

```bash
# 查看当前角色
amazing-cli role show

# 输出：
# 当前角色: 架构师 (architect)
# 权限: view:all, code:write, code:review, ...
```

## 最佳实践

### 1. 明确角色职责

**建议**：
- 每个人明确自己的角色
- 不要越权操作
- 遇到问题找对应角色

### 2. 尊重角色权限

**原则**：
- 前端不要直接修改后端代码
- 后端不要直接修改前端代码
- 开发不要直接部署生产环境

### 3. 加强角色协作

**方式**：
- 定期沟通
- 及时反馈
- 相互支持

## 常见问题

### Q1: 可以一人多角色吗？

**答**：可以，但要明确当前角色。

```bash
# 切换角色
amazing-cli role set frontend  # 开发前端
amazing-cli role set backend   # 开发后端
```

### Q2: 可以自定义角色吗？

**答**：不建议。6 种角色已经覆盖了基本需求。

### Q3: 小团队如何分配角色？

**答**：一人可以兼任多个角色。

**示例**：3 人团队
- 人员 A：架构师 + 后端
- 人员 B：前端 + QA
- 人员 C：后端 + Ops
