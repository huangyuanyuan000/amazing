# 架构师角色定义

## 1. 角色定位

架构师是 Amazing 平台的核心决策者，负责：
- 技术方案选型和决策
- 产品形态定义和审查
- 代码质量把关
- 系统架构设计
- 团队协作协调

## 2. 权限清单

### 2.1 决策权限
- ✅ 技术方案选型（从 AI 提供的 3-5 种方案中选择）
- ✅ 产品形态定义（UI/UX 设计审批）
- ✅ 任务拆分和分配
- ✅ 质量标准制定
- ✅ 上线审批

### 2.2 审查权限
- ✅ 代码审查（Code Review）
- ✅ 架构审查（Architecture Review）
- ✅ 安全审查（Security Review）
- ✅ 性能审查（Performance Review）
- ✅ 产品形态审查（UI/UX Review）

### 2.3 查看权限
- ✅ 查看所有任务
- ✅ 查看所有代码
- ✅ 查看所有日志
- ✅ 查看所有决策历史
- ✅ 查看所有审查记录

### 2.4 限制权限
- ❌ 不直接编写代码（由专业角色完成）
- ❌ 不直接部署（由运维角色完成）
- ⚠️ 可以打回不合格的代码和设计

## 3. 工作流程

### 3.1 需求阶段
1. 接收 PM 的 PRD
2. 评估技术可行性
3. 提出技术建议

### 3.2 设计阶段
1. AI 提供 3-5 种技术方案
2. 架构师评估并选择最优方案
3. AI 提供多种 UI 设计
4. 架构师定制产品形态
5. 制定质量标准

### 3.3 开发阶段
1. 拆分任务并分配给前后端
2. 监控开发进度
3. 解答技术问题
4. 协调团队协作

### 3.4 审查阶段
1. 代码审查（Code Review）
2. 架构审查（是否符合设计）
3. 产品形态审查（是否符合预期）
4. 安全审查（是否存在漏洞）
5. 性能审查（是否满足要求）

### 3.5 验收阶段
1. 功能验收
2. 性能测试
3. 安全测试
4. 批准上线或打回重做

## 4. 决策流程

### 4.1 技术方案决策
```
1. AI 分析需求
   ↓
2. AI 提供 3-5 种方案
   - 方案 A: 传统方案（稳定）
   - 方案 B: 创新方案（高效）
   - 方案 C: 折中方案（平衡）
   - ...
   ↓
3. 架构师评估
   - 技术可行性
   - 成本效益
   - 团队能力
   - 时间要求
   ↓
4. 架构师选择方案
   ↓
5. AI 在选定方案内执行
```

### 4.2 产品形态决策
```
1. AI 提供多种设计
   - 设计 A: 简洁风格
   - 设计 B: 丰富风格
   - 设计 C: 创新风格
   - ...
   ↓
2. 架构师评估
   - 用户体验
   - 品牌一致性
   - 实现难度
   - 维护成本
   ↓
3. 架构师定制产品形态
   - 选择布局
   - 定义颜色
   - 指定组件
   - 确定交互
   ↓
4. AI 在定制规范内实现
```

## 5. 审查标准

### 5.1 代码审查标准
- 代码规范（Linting）
- 代码质量（Complexity）
- 测试覆盖率（Coverage）
- 安全漏洞（Security）
- 性能问题（Performance）

### 5.2 架构审查标准
- 是否符合设计文档
- 是否遵循最佳实践
- 是否考虑扩展性
- 是否考虑可维护性

### 5.3 产品形态审查标准
- 是否符合 UI 规范
- 是否符合交互规范
- 是否符合品牌规范
- 是否符合用户体验

## 6. CLI 命令

### 6.1 架构管理
```bash
# 创建架构文档
amazing arch create <feature-name> --solution <solution> --ui-style <style>

# 定义技术方案
amazing arch solution <feature-name> --choice <choice> --reason <reason>

# 定义 UI 规范
amazing arch ui-spec <feature-name> --layout <layout> --style <style>
```

### 6.2 决策管理
```bash
# 查看待决策事项
amazing decision pending

# 做出决策
amazing decision make <decision-id> --choice <choice> --reason <reason>

# 查看决策历史
amazing decision history
```

### 6.3 审查管理
```bash
# 查看待审查任务
amazing review pending

# 开始审查
amazing review start <task-id>

# 批准任务
amazing review approve <task-id> --comment <comment>

# 拒绝任务
amazing review reject <task-id> --reason <reason>

# 查看审查历史
amazing review history
```

## 7. 与 AI 的协作模式

### 7.1 AI 的角色
- 提供多种方案供选择
- 在架构师定义的边界内执行
- 自动化重复性工作
- 提供技术建议和最佳实践

### 7.2 架构师的角色
- 做出关键决策
- 定义质量标准
- 把关产品形态
- 协调团队协作

### 7.3 协作原则
- AI 提供选项，架构师做决策
- AI 执行实现，架构师做审查
- AI 自动化流程，架构师把控方向
- AI 提供建议，架构师最终拍板

## 8. 示例场景

### 8.1 场景：新功能开发
```
1. PM 提交 PRD: "用户登录功能"
   ↓
2. AI 提供 3 种方案:
   - 方案 A: JWT + Redis (高性能)
   - 方案 B: Session + MySQL (传统)
   - 方案 C: OAuth2 (第三方)
   ↓
3. 架构师选择: 方案 A
   理由: 性能好，适合移动端
   ↓
4. AI 提供 3 种 UI 设计:
   - 设计 A: 简洁登录框
   - 设计 B: 社交登录 + 表单
   - 设计 C: 扫码登录
   ↓
5. 架构师定制: 设计 B
   要求: 主色 #007AFF，圆角 8px
   ↓
6. AI 拆分任务:
   - 前端: 登录页面 + 表单验证
   - 后端: JWT 生成 + Redis 存储
   ↓
7. 前后端并行开发
   ↓
8. 架构师审查:
   - 代码质量: ✅ 通过
   - UI 还原度: ❌ 圆角不对，打回
   ↓
9. 前端修复后重新提交
   ↓
10. 架构师验收: ✅ 批准上线
```

### 8.2 场景：Bug 修复
```
1. QA 提交 Bug: "登录失败"
   ↓
2. AI 分析日志，定位问题
   ↓
3. AI 提供 2 种修复方案:
   - 方案 A: 修改 Redis 配置
   - 方案 B: 修改 JWT 过期时间
   ↓
4. 架构师选择: 方案 B
   理由: 根本原因是过期时间太短
   ↓
5. AI 修复代码
   ↓
6. 架构师审查: ✅ 通过
   ↓
7. 批准上线
```

## 9. 最佳实践

### 9.1 决策原则
- 优先考虑长期价值，而非短期便利
- 平衡技术先进性和团队能力
- 考虑成本效益和时间要求
- 保持架构一致性

### 9.2 审查原则
- 关注关键问题，不纠结细节
- 提供建设性意见，而非批评
- 及时反馈，避免阻塞开发
- 记录决策理由，便于追溯

### 9.3 协作原则
- 信任团队，授权执行
- 明确边界，避免越权
- 及时沟通，避免误解
- 持续改进，优化流程
