# 产品形态描述指南

## 为什么需要描述产品形态？

产品形态描述是 Amazing 初始化的第一步，也是最重要的一步。

**好的产品描述**能帮助 AI：
- ✅ 准确理解产品需求
- ✅ 推荐合适的业务划分
- ✅ 推荐合适的技术栈
- ✅ 生成准确的项目结构

**不好的产品描述**会导致：
- ❌ AI 理解偏差
- ❌ 业务划分不合理
- ❌ 技术选型不匹配
- ❌ 需要大量调整

---

## 如何描述产品形态？

### 描述框架

使用 **5W1H** 框架描述产品：

1. **What**（是什么）- 产品类型和定位
2. **Why**（为什么）- 解决什么问题
3. **Who**（给谁用）- 目标用户
4. **Where**（在哪用）- 使用场景
5. **When**（什么时候）- 使用时机
6. **How**（怎么用）- 核心功能

### 描述模板

```
我要构建一个 [产品类型]，主要解决 [核心问题]。

目标用户：[用户群体]
使用场景：[场景描述]

核心功能包括：
1. [功能模块 1]
   - [子功能 1.1]
   - [子功能 1.2]
   - [子功能 1.3]

2. [功能模块 2]
   - [子功能 2.1]
   - [子功能 2.2]

3. [功能模块 3]
   - [子功能 3.1]
   - [子功能 3.2]

技术要求：
- [要求 1]
- [要求 2]
- [要求 3]

预期规模：
- 用户数：[数量]
- 并发量：[数量]
- 数据量：[数量]
```

## 示例 1：大模型管理平台

### 完整描述

```
我要构建一个企业级大模型管理平台，主要解决企业在使用大模型时
面临的资源管理、数据管理、训练管理和服务部署等问题。

目标用户：
- AI 工程师（训练和调优模型）
- 算法研究员（实验和研究）
- 应用开发者（使用模型 API）
- 平台管理员（管理资源和用户）

使用场景：
- 企业内部 AI 平台
- 多团队共享 GPU 资源
- 模型训练和推理服务

核心功能包括：

1. 算力资源管理
   - GPU/CPU 资源池管理
   - 资源调度和分配
   - 资源监控和告警
   - 成本统计和优化
   - 多集群管理

2. 数据管理
   - 数据集上传和管理
   - 数据标注工具
   - 数据版本控制
   - 数据质量检查
   - 数据权限管理

3. 模型训练
   - 训练任务创建和管理
   - 分布式训练支持
   - 超参数调优
   - 实验跟踪和对比
   - 训练日志和可视化

4. 模型服务
   - 模型版本管理
   - 模型部署和推理
   - API 服务管理
   - 性能监控和优化
   - 流量控制和限流

5. 用户和权限
   - 多租户支持
   - 角色权限管理
   - 审计日志
   - 配额管理
   - SSO 集成

技术要求：
- 高性能（支持大规模并发）
- 高可用（99.9% 可用性）
- 易扩展（支持水平扩展）
- 安全可靠（数据加密、权限控制）

预期规模：
- 用户数：100-1000 人
- GPU 数量：50-200 张
- 并发训练任务：20-50 个
- API QPS：1000-5000
```

### AI 分析结果

```
✨ 产品分析结果：

产品类型：AI/ML 平台
核心模块：5 个（算力、数据、训练、服务、用户）
复杂度：高
推荐团队规模：10-15 人
推荐开发周期：6-9 个月

推荐业务划分方案：

方案 A（推荐）：按资源类型划分
├── 算力平台 Agent - GPU/CPU 资源管理
├── 数据 Agent - 数据集管理
├── 训练 Agent - 训练任务管理
└── 模型服务 Agent - 模型部署和服务

方案 B：按功能流程划分
├── 资源 Agent - 算力和数据资源
├── 工作流 Agent - 训练和推理流程
└── 服务 Agent - API 和监控服务

方案 C：按用户角色划分
├── 管理员 Agent - 平台管理功能
├── 开发者 Agent - 训练和开发功能
└── 用户 Agent - 使用和查询功能
```

## 示例 2：电商平台

### 完整描述

```
我要构建一个 B2C 电商平台，主要解决中小企业在线销售的需求。

目标用户：
- 消费者（浏览和购买商品）
- 商家（管理商品和订单）
- 平台运营（管理平台和数据）

使用场景：
- 在线购物
- 商家入驻
- 平台运营

核心功能包括：

1. 商品管理
   - 商品发布和编辑
   - 商品分类和标签
   - 商品搜索和筛选
   - 商品推荐

2. 订单管理
   - 购物车
   - 下单和支付
   - 订单跟踪
   - 退款和售后

3. 支付管理
   - 多种支付方式
   - 支付安全
   - 对账和结算

4. 物流管理
   - 物流跟踪
   - 配送管理
   - 物流商对接

5. 用户管理
   - 用户注册和登录
   - 用户信息管理
   - 会员体系
   - 积分和优惠券

技术要求：
- 高并发（支持秒杀活动）
- 高可用（24/7 在线）
- 数据安全（支付和用户数据）

预期规模：
- 用户数：10 万 - 100 万
- 商品数：1 万 - 10 万
- 日订单量：1000 - 10000
```

### AI 分析结果

```
✨ 产品分析结果：

产品类型：电商平台
核心模块：5 个（商品、订单、支付、物流、用户）
复杂度：中高
推荐团队规模：8-12 人
推荐开发周期：4-6 个月

推荐业务划分方案：

方案 A（推荐）：按业务流程划分
├── 商品 Agent - 商品管理
├── 订单 Agent - 订单和购物车
├── 支付 Agent - 支付和结算
└── 物流 Agent - 配送和跟踪

方案 B：按用户角色划分
├── 消费者 Agent - 购物功能
├── 商家 Agent - 商家管理
└── 平台 Agent - 平台运营

方案 C：按技术领域划分
├── 交易 Agent - 订单和支付
├── 商品 Agent - 商品和搜索
└── 服务 Agent - 物流和售后
```

## 描述要点

### 1. 要具体，不要抽象

**好的描述**：
```
✅ 支持 GPU/CPU 资源池管理，包括资源分配、调度、监控
✅ 支持数据集上传（最大 100GB）、在线标注、版本控制
✅ 支持分布式训练（最多 8 卡并行）
```

**不好的描述**：
```
❌ 做一个 AI 平台
❌ 管理资源
❌ 处理数据
```

### 2. 要分层，不要平铺

**好的描述**：
```
✅ 1. 算力资源管理
      - GPU/CPU 资源池
      - 资源调度
      - 资源监控
```

**不好的描述**：
```
❌ GPU 管理、CPU 管理、资源调度、资源监控、...
```

### 3. 要量化，不要模糊

**好的描述**：
```
✅ 用户数：100-1000 人
✅ 并发量：1000-5000 QPS
✅ 数据量：10TB-100TB
```

**不好的描述**：
```
❌ 很多用户
❌ 高并发
❌ 大数据量
```

### 4. 要完整，不要遗漏

**必须包含**：
- ✅ 产品类型和定位
- ✅ 目标用户
- ✅ 核心功能（3-5 个模块）
- ✅ 技术要求
- ✅ 预期规模

**可选包含**：
- 使用场景
- 竞品对比
- 特殊需求

## 常见问题

### Q1: 描述太长会怎样？

**答**：不会有问题，描述越详细越好。AI 会提取关键信息。

### Q2: 不确定某些功能怎么办？

**答**：可以标注为"可选"或"未来规划"，先描述确定的部分。

### Q3: 技术要求不清楚怎么办？

**答**：可以描述业务要求（如"支持秒杀"），AI 会推荐技术方案。

### Q4: 可以修改描述吗？

**答**：可以。初始化后可以调整业务 Agent 和技术栈。

## 检查清单

在提交产品描述前，检查以下内容：

- [ ] 产品类型明确
- [ ] 目标用户清晰
- [ ] 核心功能完整（3-5 个模块）
- [ ] 每个模块有子功能说明
- [ ] 技术要求明确
- [ ] 预期规模量化
- [ ] 描述结构清晰
- [ ] 没有歧义和模糊表述
