# AI 自动化模式说明

## 概述

Amazing 平台支持两种 AI 自动化模式，**架构师可以根据需求维度灵活切换**：

1. **全自动 AI 模式 (Full-Auto)**：AI 独立完成任务，遇到问题时人工可介入
2. **半自动 AI 模式 (Semi-Auto)**：AI 负责生成方案，人工负责决策

## 重要说明

### 权限控制
- ⚠️ **只有架构师有权限切换模式**
- 模式切换需要关联具体需求ID
- 所有切换操作都会被记录和审计

### 切换维度
- 模式切换是**按需求维度**进行的
- 不同需求可以使用不同的模式
- 同一需求的不同 Agent 也可以使用不同模式

## 模式对比

| 特性 | 全自动模式 | 半自动模式 |
|------|-----------|-----------|
| **AI 角色** | 执行者 | 辅助者 |
| **人工角色** | 监督者 | 决策者 |
| **决策权** | AI 自主决策 | 人工决策 |
| **执行权** | AI 自动执行 | 人工批准后执行 |
| **适用场景** | 成熟流程、低风险任务 | 新项目、高风险任务 |
| **效率** | 高（无需等待） | 中（需要审批） |
| **可控性** | 中（关键节点检查） | 高（全程可控） |
| **切换权限** | 仅架构师 | 仅架构师 |

## 一、全自动 AI 模式 (Full-Auto)

### 1.1 核心特性

#### 特性 1：AI 自主决策和执行
- AI 根据需求自动选择技术方案
- AI 自动生成代码并提交
- AI 自动运行测试并修复问题
- AI 自动部署到开发环境

#### 特性 2：自动推进工作流
```
需求输入 → AI 分析 → AI 设计 → AI 开发 → AI 测试 → AI 部署
```
整个流程自动推进，无需人工干预。

#### 特性 3：关键节点自动检查点
在以下关键节点，AI 会暂停并通知人工：
- **架构设计**：重大架构变更
- **技术选型**：引入新技术栈
- **安全审查**：发现安全风险
- **生产部署**：部署到生产环境

#### 特性 4：异常时暂停等待人工介入
当 AI 遇到以下情况时，会自动暂停：
- **Critical 错误**：系统崩溃、数据丢失
- **High 错误**：功能无法实现、测试失败
- **Medium 错误**：性能问题、代码质量问题（记录日志，继续执行）
- **Low 错误**：代码风格问题（仅记录日志）

#### 特性 5：人工可随时接管控制
人工可以随时使用以下命令接管控制：
- `pause`：暂停 AI 执行
- `resume`：恢复 AI 执行
- `rollback`：回滚到上一个稳定状态
- `approve`：批准 AI 的方案
- `reject`：拒绝 AI 的方案

### 1.2 工作流程

#### 场景 1：功能开发

```
1. 用户输入需求
   ↓
2. AI 自动分析需求
   ├── 生成 PRD
   ├── 设计架构
   └── 选择技术栈
   ↓
3. [检查点] 架构设计
   ├── 通知人工审查（5分钟超时）
   ├── 人工批准 → 继续
   └── 超时 → 自动继续
   ↓
4. AI 自动开发
   ├── 生成前端代码
   ├── 生成后端代码
   └── 生成测试代码
   ↓
5. AI 自动测试
   ├── 运行单元测试
   ├── 运行集成测试
   └── 修复失败的测试
   ↓
6. [检查点] 安全审查
   ├── 通知人工审查（10分钟超时）
   ├── 人工批准 → 继续
   └── 超时 → 回滚
   ↓
7. AI 自动部署
   ├── 部署到开发环境
   └── 运行冒烟测试
   ↓
8. [检查点] 生产部署
   ├── 通知人工审查（15分钟超时）
   ├── 人工批准 → 部署
   └── 超时 → 取消
```

#### 场景 2：Bug 修复

```
1. 用户报告 Bug
   ↓
2. AI 自动复现 Bug
   ├── 分析日志
   ├── 定位问题代码
   └── 生成修复方案
   ↓
3. AI 自动修复
   ├── 修改代码
   ├── 运行测试
   └── 验证修复
   ↓
4. [检查点] 代码审查
   ├── 通知人工审查（5分钟超时）
   ├── 人工批准 → 继续
   └── 超时 → 继续
   ↓
5. AI 自动部署
   ├── 热修复部署
   └── 监控运行状态
```

### 1.3 人工介入时机

#### 主动介入
人工可以随时主动介入：
```bash
# 暂停 AI 执行
$ amazing mode pause

# 查看当前状态
$ amazing status

# 接管控制
$ amazing mode manual

# 恢复 AI 执行
$ amazing mode resume
```

#### 被动介入
AI 在以下情况会请求人工介入：
1. **关键决策点**：架构设计、技术选型、生产部署
2. **异常情况**：Critical/High 错误
3. **超时情况**：检查点超时

## 二、半自动 AI 模式 (Semi-Auto)

### 2.1 核心特性

#### 特性 1：AI 生成方案和建议
- AI 分析需求并生成多种技术方案
- AI 提供每种方案的优缺点对比
- AI 推荐最佳方案（但不决策）
- AI 生成详细的实施计划

#### 特性 2：人工审核和决策
- 人工审查 AI 生成的方案
- 人工选择最终采用的方案
- 人工批准每个关键步骤
- 人工决定是否继续执行

#### 特性 3：人工可提供建议和想法
- 人工可以对 AI 的方案提出修改意见
- 人工可以提供新的需求和约束
- 人工可以指定特定的技术栈
- 人工可以调整实施优先级

#### 特性 4：AI 根据反馈优化方案
- AI 根据人工反馈重新生成方案
- AI 学习人工的偏好和决策模式
- AI 逐步提高方案的准确性
- AI 减少人工审核的工作量

#### 特性 5：关键决策点必须人工确认
以下决策点必须人工确认：
- **架构设计**：系统架构、模块划分
- **技术选型**：技术栈、第三方组件
- **代码合并**：合并到主分支
- **部署发布**：部署到生产环境
- **安全变更**：权限、加密、认证

### 2.3 反馈机制

#### 反馈类型

1. **评论 (Comment)**
   ```

   ```

2. **建议 (Suggestion)**
   ```
   人工: "建议增加缓存层，使用 Redis"
   AI: "好的，我已经在架构中增加了 Redis 缓存层，并更新了技术方案"
   ```

3. **需求 (Requirement)**
   ```
   人工: "必须支持分布式部署"
   AI: "好的，我已经将架构调整为支持分布式部署，并更新了部署方案"
   ```

#### 反馈流程

```
1. AI 生成方案
   ↓

   ├── 批准 → 执行
   ├── 评论 → AI 优化
   ├── 建议 → AI 增强
   └── 需求 → AI 重新设计
   ↓
3. AI 根据反馈优化
   ↓
4. 人工再次审查
   ↓
5. 循环直到批准
```

## 三、模式切换

### 3.1 权限要求

#### 谁可以切换模式？
- ✅ **架构师 (Architect)**：拥有完整的模式切换权限
- ❌ **其他角色**：无权切换模式

#### 为什么只有架构师可以切换？
1. **技术决策权**：架构师负责整体技术方案，有权决定自动化程度
2. **风险评估能力**：架构师能够评估不同模式的风险和收益
3. **全局视角**：架构师了解项目整体情况，能做出最优决策
4. **责任明确**：模式切换影响重大，需要明确责任人

### 3.2 切换命令

```bash
# 1. 首先切换到架构师角色
$ python3 scripts/amazing-cli.py role set architect

# 2. 查看当前模式
$ python3 scripts/mode_cli.py mode show

# 3. 切换到全自动模式（需要提供需求ID）
$ python3 scripts/mode_cli.py mode set full-auto -r REQ-001 --reason "成熟功能，低风险"

# 4. 切换到半自动模式
$ python3 scripts/mode_cli.py mode set semi-auto -r REQ-002 --reason "新项目，需要人工把关"

# 5. 查看切换历史
$ python3 scripts/mode_cli.py mode history

# 6. 查看模式配置
$ python3 scripts/mode_cli.py mode config
```

### 3.3 切换流程

#### 完整切换流程

```
1. 架构师接收需求
   ├── 需求ID: REQ-001
   ├── 需求类型: 功能开发
   └── 风险等级: 低
   ↓
2. 架构师评估需求
   ├── 技术复杂度: 低
   ├── 业务风险: 低
   ├── 团队熟悉度: 高
   └── 决策: 使用全自动模式
   ↓
3. 架构师切换模式
   $ python3 scripts/mode_cli.py mode set full-auto \
     -r REQ-001 \
     --reason "成熟功能，团队熟悉，风险可控"
   ↓
4. 系统记录切换
   ├── 时间戳
   ├── 操作者: architect
   ├── 需求ID: REQ-001
   ├── 切换原因
   └── 模式: semi-auto → full-auto
   ↓
5. AI 按新模式执行
   ├── 全自动模式: AI 自主执行
   └── 关键节点: 架构师审查
```

#### 权限验证流程

```
用户执行切换命令
   ↓
系统检查用户角色
   ├── 是架构师? → 继续
   └── 不是架构师? → 拒绝
       ├── 提示: "权限不足：只有架构师可以切换模式"
       └── 建议: "请先切换到架构师角色"
   ↓
验证需求ID
   ├── 提供了需求ID? → 继续
   └── 未提供? → 警告（可选）
   ↓
执行模式切换
   ├── 更新配置文件
   ├── 记录切换历史
   └── 记录审计日志
   ↓
返回切换结果
```

### 3.4 切换时机

#### 建议使用全自动模式的场景
- ✅ 成熟的业务流程
- ✅ 低风险的功能开发
- ✅ Bug 修复
- ✅ 代码重构
- ✅ 性能优化
- ✅ 团队熟悉的技术栈

#### 建议使用半自动模式的场景
- ✅ 新项目启动
- ✅ 架构设计
- ✅ 技术选型
- ✅ 安全相关变更
- ✅ 生产环境部署
- ✅ 不熟悉的技术栈

### 3.5 按需求维度切换

#### 什么是按需求维度切换？
- 每个需求可以独立设置模式
- 不同需求可以使用不同模式
- 模式与需求ID绑定

#### 示例：多需求并行

```bash
# 需求 REQ-001: 用户登录功能（成熟功能）
$ python3 scripts/mode_cli.py mode set full-auto -r REQ-001

# 需求 REQ-002: 支付系统（高风险）
$ python3 scripts/mode_cli.py mode set semi-auto -r REQ-002

# 需求 REQ-003: 数据报表（低风险）
$ python3 scripts/mode_cli.py mode set full-auto -r REQ-003
```

#### 按 Agent 维度切换

```bash
# Common Agent 使用全自动模式（成熟模块）
$ python3 scripts/mode_cli.py agent mode common full-auto -r REQ-001

# Training Agent 使用半自动模式（复杂逻辑）
$ python3 scripts/mode_cli.py agent mode training semi-auto -r REQ-001
```

## 四、最佳实践

### 4.1 全自动模式最佳实践

1. **设置合理的检查点**
   - 关键决策点必须设置检查点
   - 检查点超时时间要合理
   - 超时策略要根据风险设置

2. **配置错误处理策略**
   - Critical/High 错误必须暂停
   - Medium 错误可以继续执行
   - Low 错误仅记录日志

3. **保持人工监督**
   - 定期查看 AI 执行状态
   - 及时响应 AI 的通知
   - 必要时主动介入

### 4.2 半自动模式最佳实践

1. **提供清晰的反馈**
   - 明确指出问题所在
   - 提供具体的修改建议
   - 说明修改的理由

2. **及时审批**
   - 尽快审查 AI 生成的方案
   - 避免长时间阻塞流程
   - 设置合理的超时时间

3. **积累经验**
   - 记录常见的决策模式
   - 总结最佳实践
   - 优化审批流程

## 五、常见问题

### Q1: 如何选择合适的模式？

**A**: 根据以下因素选择：
- **项目成熟度**：新项目用半自动，成熟项目用全自动
- **风险等级**：高风险用半自动，低风险用全自动
- **团队经验**：经验丰富用全自动，新手用半自动

### Q2: 全自动模式下如何保证质量？

**A**: 通过以下机制保证质量：
- 关键节点设置检查点
- 自动化测试覆盖
- 代码质量扫描
- 安全漏洞检测
- 人工可随时介入

### Q3: 半自动模式会不会太慢？

**A**: 可以通过以下方式提高效率：
- 设置合理的审批超时
- 批量审批相似的任务
- 积累决策模式，减少审批次数
- 对低风险任务使用全自动模式

### Q4: 可以混合使用两种模式吗？

**A**: 可以，支持以下混合方式：
- 不同 Agent 使用不同模式
- 不同场景使用不同模式
- 不同环境使用不同模式（开发环境全自动，生产环境半自动）
