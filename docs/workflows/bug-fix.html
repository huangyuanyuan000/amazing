1 问题发现与报告
运维工程师
测试工程师
用户

通过监控告警或用户反馈发现问题。

$ claude code
> 提交 Bug 报告：用户登录失败
> 严重程度：高
> 影响范围：所有用户
> 错误信息：500 Internal Server Error
🚨 严重程度分级：
  • P0 - 紧急：核心功能不可用，影响所有用户
  • P1 - 高：重要功能异常，影响部分用户
  • P2 - 中：次要功能问题，影响较小
  • P3 - 低：体验优化，不影响功能

3 回滚决策
运维工程师

根据严重程度决定是否立即回滚。

🚨 P0/P1 级别 Bug - 立即回滚：
$ make rollback

# 或指定版本
$ make rollback VERSION=v1.2.2
⚠️ P2/P3 级别 Bug - 进入修复流程：

不影响核心功能，可以通过修复代码解决。

🔄 回滚操作指南

1. 查看可回滚版本

$ make list-versions

# 输出示例：
# v1.2.3 (current) - 2026-03-07 10:00
# v1.2.2 (stable) - 2026-03-06 15:00
# v1.2.1 (stable) - 2026-03-05 12:00

2. 回滚到指定版本

# 回滚到上一版本
$ make rollback

# 回滚到指定版本
$ make rollback VERSION=v1.2.2

3. 验证回滚结果

$ make verify-rollback

# 检查：
# 1. 服务是否正常启动
# 2. 健康检查是否通过
# 3. 关键接口是否可用
# 4. 数据库连接是否正常

4. 回滚失败处理

🚨 如果回滚失败：
  • 立即通知团队
  • 查看回滚日志：make rollback-logs
  • 尝试回滚到更早的稳定版本
  • 必要时启用备用系统

✨ 最佳实践

1. 预防措施

  • 代码必须通过 Review Agent 审核
  • 测试覆盖率必须 > 80%
  • 关键功能必须有集成测试
  • 上线前必须在预发环境验证

2. 快速响应

  • 建立 7x24 小时值班制度
  • 配置告警通知（钉钉/企业微信/邮件）
  • P0 级别 Bug 15 分钟内响应
  • P1 级别 Bug 1 小时内响应

3. 问题复盘

  • 每次 P0/P1 Bug 必须复盘
  • 分析根本原因
  • 制定改进措施
  • 更新 Agent 知识库
