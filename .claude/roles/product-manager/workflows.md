# 产品经理专属工作流

## 1. 需求分析工作流

### 触发条件
- 用户反馈收集
- 业务目标变更
- 市场需求调研
- 竞品分析发现

### 工作流程
```
需求收集 → 需求分析 → 优先级排序 → PRD 编写 → 需求评审
```

### 详细步骤

#### 1.1 需求收集
- **输入**: 用户反馈、业务目标、市场调研
- **输出**: 需求清单
- **时间**: 1-3 天
- **检查点**:
  - [ ] 需求来源明确
  - [ ] 需求描述清晰
  - [ ] 用户痛点识别
  - [ ] 业务价值评估
- **收集渠道**:
  - 用户访谈
  - 问卷调查
  - 数据分析
  - 客服反馈
  - 竞品分析
  - 内部建议

#### 1.2 需求分析
- **输入**: 需求清单
- **输出**: 需求分析报告
- **时间**: 2-3 天
- **分析维度**:
  - **用户价值**: 解决什么问题？影响多少用户？
  - **业务价值**: 对业务指标的影响？ROI 如何？
  - **技术可行性**: 技术难度？开发成本？
  - **资源投入**: 需要多少人力？多长时间？
  - **风险评估**: 有哪些风险？如何应对？
- **检查点**:
  - [ ] 需求合理性验证
  - [ ] 用户场景清晰
  - [ ] 价值量化评估
  - [ ] 可行性确认

#### 1.3 优先级排序
- **输入**: 需求分析报告
- **输出**: 优先级排序清单
- **时间**: 1 天
- **排序方法**: RICE 模型
  - **Reach (影响范围)**: 影响多少用户？
  - **Impact (影响程度)**: 对用户的影响有多大？
  - **Confidence (信心指数)**: 有多大把握？
  - **Effort (工作量)**: 需要多少工作量？
  - **得分**: (R × I × C) / E
- **优先级分类**:
  - P0: 必须做（核心功能、严重 Bug）
  - P1: 应该做（重要功能、用户强需求）
  - P2: 可以做（优化改进、次要功能）
  - P3: 暂不做（低价值、高成本）

#### 1.4 PRD 编写
- **输入**: 优先级排序清单
- **输出**: PRD 文档
- **时间**: 3-5 天
- **详见**: [PRD 编写工作流](#2-prd-编写工作流)

#### 1.5 需求评审
- **输入**: PRD 文档
- **输出**: 评审意见、最终 PRD
- **时间**: 1 天
- **详见**: [需求评审工作流](#3-需求评审工作流)

## 2. PRD 编写工作流

### 触发条件
- 需求分析完成
- 优先级确定
- 准备进入开发

### 工作流程
```
背景分析 → 目标定义 → 功能设计 → 原型设计 → 验收标准 → 评审修订
```

### 详细步骤

#### 2.1 背景分析
- **时间**: 1 天
- **输出**: PRD 背景部分
- **内容**:
  - **业务背景**: 为什么要做这个需求？
  - **用户痛点**: 用户遇到了什么问题？
  - **现状分析**: 当前是什么情况？
  - **竞品分析**: 竞品是怎么做的？
  - **数据支撑**: 有哪些数据支持？
- **检查点**:
  - [ ] 背景描述清晰
  - [ ] 痛点真实存在
  - [ ] 数据充分支撑

#### 2.2 目标定义
- **时间**: 0.5 天
- **输出**: PRD 目标部分
- **内容**:
  - **产品目标**: 这个功能要达到什么目的？
  - **用户目标**: 用户能获得什么价值？
  - **业务目标**: 对业务指标的预期影响？
  - **成功指标**: 如何衡量成功？
- **目标原则**: SMART
  - Specific (具体的)
  - Measurable (可衡量的)
  - Achievable (可实现的)
  - Relevant (相关的)
  - Time-bound (有时限的)
- **检查点**:
  - [ ] 目标明确可衡量
  - [ ] 成功指标可量化
  - [ ] 目标与业务对齐

#### 2.3 功能设计
- **时间**: 2-3 天
- **输出**: PRD 功能设计部分
- **内容**:
  - **功能列表**: 包含哪些功能？
  - **功能优先级**: P0/P1/P2 分级
  - **用户场景**: 用户如何使用？
  - **功能流程**: 业务流程图
  - **交互说明**: 页面交互逻辑
  - **异常处理**: 异常情况如何处理？
- **设计原则**:
  - 用户视角（不是技术视角）
  - 场景化描述
  - 逻辑完整
  - 边界清晰
- **检查点**:
  - [ ] 功能描述完整
  - [ ] 用户场景清晰
  - [ ] 流程逻辑正确
  - [ ] 异常处理完善

#### 2.4 原型设计
- **时间**: 2-3 天
- **输出**: 交互原型
- **工具**: Figma / Axure / Sketch
- **内容**:
  - **页面结构**: 页面布局和层级
  - **交互流程**: 页面跳转和交互
  - **元素说明**: 按钮、输入框等元素
  - **状态说明**: 不同状态的展示
- **原型类型**:
  - 低保真原型（快速验证）
  - 高保真原型（详细设计）
- **检查点**:
  - [ ] 页面结构合理
  - [ ] 交互流程顺畅
  - [ ] 元素说明清晰
  - [ ] 覆盖所有状态

#### 2.5 验收标准
- **时间**: 1 天
- **输出**: PRD 验收标准部分
- **内容**:
  - **功能验收**: 每个功能的验收标准
  - **性能验收**: 响应时间、并发量等
  - **兼容性验收**: 浏览器、设备兼容
  - **安全验收**: 权限、数据安全
  - **体验验收**: 易用性、流畅度
- **标准原则**:
  - 可验证（明确的判断标准）
  - 可量化（具体的数值指标）
  - 可测试（可以通过测试验证）
- **检查点**:
  - [ ] 每个功能都有验收标准
  - [ ] 标准明确可验证
  - [ ] 覆盖功能、性能、体验

#### 2.6 评审修订
- **时间**: 1-2 天
- **活动**: 需求评审会议
- **详见**: [需求评审工作流](#3-需求评审工作流)

## 3. 需求评审工作流

### 触发条件
- PRD 初稿完成
- 准备进入开发
- 重大需求变更

### 工作流程
```
评审准备 → 评审会议 → 意见收集 → PRD 修订 → 最终确认
```

### 详细步骤

#### 3.1 评审准备
- **时间**: 0.5 天
- **准备内容**:
  - PRD 文档（提前 2 天发送）
  - 原型文件
  - 评审议程
  - 会议室预定
  - 参会人员确认
- **参会人员**:
  - 产品经理（主持）
  - 架构师（必须）
  - 前端开发（必须）
  - 后端开发（必须）
  - 测试工程师（必须）
  - 运维工程师（可选）
  - UI 设计师（可选）
- **检查点**:
  - [ ] PRD 文档完整
  - [ ] 提前发送文档
  - [ ] 参会人员确认

#### 3.2 评审会议
- **时间**: 1-2 小时
- **会议流程**:
  1. **需求讲解** (20 分钟)
     - 背景和目标
     - 功能设计
     - 原型演示
  2. **技术评审** (30 分钟)
     - 架构师评估技术方案
     - 开发评估工作量
     - 识别技术风险
  3. **测试评审** (15 分钟)
     - 测试工程师评估测试范围
     - 识别测试难点
  4. **讨论和质疑** (20 分钟)
     - 需求合理性讨论
     - 技术可行性讨论
     - 优先级讨论
  5. **总结和决策** (10 分钟)
     - 确认是否通过
     - 明确修改意见
     - 确定开发排期

#### 3.3 意见收集
- **时间**: 会议中
- **收集内容**:
  - 需求理解问题
  - 功能设计建议
  - 技术实现建议
  - 风险和疑虑
  - 工作量评估
- **记录方式**:
  - 会议纪要
  - 问题清单
  - 修改建议清单

#### 3.4 PRD 修订
- **时间**: 1-2 天
- **修订内容**:
  - 根据评审意见修改 PRD
  - 补充遗漏的内容
  - 澄清模糊的描述
  - 调整不合理的设计
- **检查点**:
  - [ ] 所有评审意见已处理
  - [ ] 修改内容已标注
  - [ ] 版本号已更新

#### 3.5 最终确认
- **时间**: 0.5 天
- **确认方式**:
  - 发送修订后的 PRD
  - 关键人员确认
  - 存档最终版本
- **检查点**:
  - [ ] 架构师确认
  - [ ] 技术 Leader 确认
  - [ ] 测试负责人确认
  - [ ] PRD 已存档

## 4. 功能验收工作流

### 触发条件
- 开发完成
- 测试通过
- 准备上线

### 工作流程
```
验收准备 → 功能测试 → 验收确认 → 问题反馈 → 上线决策
```

### 详细步骤

#### 4.1 验收准备
- **时间**: 0.5 天
- **准备内容**:
  - 验收环境确认
  - 验收用例准备
  - 测试数据准备
  - 验收时间安排
- **检查点**:
  - [ ] 验收环境可用
  - [ ] 验收用例完整
  - [ ] 测试数据充分

#### 4.2 功能测试
- **时间**: 1-2 天
- **测试内容**:
  - **功能完整性**: 所有功能是否实现？
  - **功能正确性**: 功能是否符合 PRD？
  - **交互体验**: 交互是否流畅？
  - **异常处理**: 异常情况是否处理？
  - **性能表现**: 响应速度是否满足？
  - **兼容性**: 不同浏览器/设备是否正常？
- **测试方法**:
  - 按照验收用例逐项测试
  - 模拟真实用户场景
  - 尝试边界情况
  - 尝试异常操作
- **记录内容**:
  - 测试结果（通过/不通过）
  - 发现的问题
  - 改进建议

#### 4.3 验收确认
- **时间**: 0.5 天
- **确认标准**:
  - [ ] 所有 P0 功能通过
  - [ ] 所有 P1 功能通过
  - [ ] 无严重 Bug
  - [ ] 性能满足要求
  - [ ] 体验符合预期
- **确认方式**:
  - 填写验收报告
  - 签署验收确认
  - 通知相关方

#### 4.4 问题反馈
- **时间**: 实时
- **问题分类**:
  - **阻塞问题**: 必须修复才能上线
  - **重要问题**: 应该修复但不阻塞上线
  - **优化建议**: 后续迭代优化
- **反馈方式**:
  - 创建 Bug 单
  - 标注优先级
  - 指定责任人
  - 跟踪修复进度

#### 4.5 上线决策
- **时间**: 0.5 天
- **决策依据**:
  - 验收结果
  - Bug 情况
  - 业务紧急度
  - 风险评估
- **决策选项**:
  - **通过上线**: 验收通过，无阻塞问题
  - **修复后上线**: 有阻塞问题，修复后上线
  - **延期上线**: 问题严重，需要延期
  - **分阶段上线**: 部分功能先上线
- **输出**:
  - 上线决策报告
  - 上线计划

## 5. 产品迭代工作流

### 触发条件
- 功能上线后
- 定期迭代评审
- 用户反馈积累
- 数据分析发现

### 工作流程
```
数据分析 → 问题识别 → 方案设计 → 评审 → 开发跟进 → 效果评估
```

### 详细步骤

#### 5.1 数据分析
- **时间**: 2-3 天
- **分析维度**:
  - **使用数据**:
    - 功能使用率
    - 用户活跃度
    - 转化率
    - 留存率
  - **性能数据**:
    - 响应时间
    - 错误率
    - 崩溃率
  - **用户反馈**:
    - 用户评价
    - 客服反馈
    - 用户访谈
- **分析方法**:
  - 漏斗分析
  - 留存分析
  - 用户分群
  - A/B 测试结果
- **输出**:
  - 数据分析报告
  - 问题清单

#### 5.2 问题识别
- **时间**: 1 天
- **问题分类**:
  - **功能问题**: 功能缺失、功能不完善
  - **体验问题**: 交互不流畅、操作复杂
  - **性能问题**: 响应慢、卡顿
  - **Bug**: 功能异常、数据错误
- **优先级评估**:
  - 影响范围
  - 严重程度
  - 修复成本
  - 业务价值
- **输出**:
  - 问题清单（按优先级排序）

#### 5.3 方案设计
- **时间**: 2-3 天
- **设计内容**:
  - 优化方案
  - 功能改进
  - 新功能设计
- **设计原则**:
  - 数据驱动
  - 用户导向
  - 小步快跑
  - 快速验证
- **输出**:
  - 优化方案文档
  - 简化版 PRD

#### 5.4 评审
- **时间**: 1 天
- **评审内容**:
  - 方案合理性
  - 技术可行性
  - 工作量评估
  - 优先级确认
- **参与者**:
  - 产品经理
  - 架构师
  - 开发团队
- **输出**:
  - 评审通过的方案
  - 开发排期

#### 5.5 开发跟进
- **时间**: 持续
- **跟进内容**:
  - 开发进度
  - 技术问题
  - 需求变更
  - 风险管理
- **跟进方式**:
  - 每日站会
  - 定期同步
  - 问题解答
  - 需求澄清

#### 5.6 效果评估
- **时间**: 上线后 1-2 周
- **评估内容**:
  - 数据指标变化
  - 用户反馈
  - 业务目标达成
  - 经验总结
- **评估方法**:
  - 对比优化前后数据
  - A/B 测试结果分析
  - 用户访谈
- **输出**:
  - 效果评估报告
  - 经验总结
  - 后续优化建议

## 6. 用户研究工作流

### 触发条件
- 新功能设计前
- 用户体验优化
- 产品方向调整
- 定期用户调研

### 工作流程
```
研究目标 → 研究设计 → 数据收集 → 数据分析 → 洞察输出 → 应用落地
```

### 详细步骤

#### 6.1 研究目标
- **时间**: 0.5 天
- **明确内容**:
  - 研究问题
  - 研究目的
  - 预期产出
- **示例**:
  - 了解用户对某功能的使用习惯
  - 验证新功能的用户需求
  - 发现用户痛点

#### 6.2 研究设计
- **时间**: 1 天
- **设计内容**:
  - 研究方法选择
  - 用户样本选择
  - 研究工具准备
  - 时间计划
- **研究方法**:
  - **定性研究**: 用户访谈、可用性测试
  - **定量研究**: 问卷调查、数据分析
  - **混合研究**: 定性 + 定量

#### 6.3 数据收集
- **时间**: 3-5 天
- **收集方式**:
  - 用户访谈（1对1 深度访谈）
  - 问卷调查（在线问卷）
  - 可用性测试（观察用户使用）
  - 数据埋点（行为数据）
- **样本要求**:
  - 样本量充足
  - 样本代表性
  - 样本多样性

#### 6.4 数据分析
- **时间**: 2-3 天
- **分析方法**:
  - 定性分析（主题分析、内容分析）
  - 定量分析（统计分析、相关分析）
  - 用户画像
  - 用户旅程地图
- **输出**:
  - 数据分析报告

#### 6.5 洞察输出
- **时间**: 1 天
- **洞察内容**:
  - 用户需求
  - 用户痛点
  - 用户行为模式
  - 机会点识别
- **输出**:
  - 用户研究报告
  - 设计建议

#### 6.6 应用落地
- **时间**: 持续
- **应用方式**:
  - 指导产品设计
  - 优化用户体验
  - 调整产品策略
  - 分享给团队

## 7. 竞品分析工作流

### 触发条件
- 新功能设计前
- 产品策略调整
- 定期竞品监控
- 市场变化

### 工作流程
```
竞品选择 → 信息收集 → 功能对比 → 优劣分析 → 策略建议
```

### 详细步骤

#### 7.1 竞品选择
- **时间**: 0.5 天
- **选择标准**:
  - 直接竞品（同类产品）
  - 间接竞品（替代方案）
  - 潜在竞品（新兴产品）
- **选择数量**: 3-5 个

#### 7.2 信息收集
- **时间**: 2-3 天
- **收集内容**:
  - 产品定位
  - 核心功能
  - 用户体验
  - 商业模式
  - 市场表现
  - 用户评价
- **收集方式**:
  - 产品体验
  - 官网资料
  - 用户评论
  - 行业报告
  - 新闻资讯

#### 7.3 功能对比
- **时间**: 1-2 天
- **对比维度**:
  - 功能完整性
  - 功能创新性
  - 交互体验
  - 性能表现
  - 视觉设计
- **对比方法**:
  - 功能对比表
  - 体验对比
  - 截图对比

#### 7.4 优劣分析
- **时间**: 1 天
- **分析内容**:
  - 竞品优势
  - 竞品劣势
  - 我方优势
  - 我方劣势
  - 差异化机会
- **分析工具**:
  - SWOT 分析
  - 四象限分析

#### 7.5 策略建议
- **时间**: 1 天
- **建议内容**:
  - 功能优化建议
  - 差异化策略
  - 创新方向
  - 风险提示
- **输出**:
  - 竞品分析报告
