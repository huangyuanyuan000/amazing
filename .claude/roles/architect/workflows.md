# 架构师专属工作流

## 1. 架构设计工作流

### 触发条件
- 新项目启动
- 重大功能开发
- 架构重构需求
- 技术栈升级

### 工作流程
```
需求输入 → 需求分析 → 架构设计 → 技术选型 → 方案评审 → 文档输出 → 实施指导
```

### 详细步骤

#### 1.1 需求分析
- **输入**: PRD 文档、业务需求
- **输出**: 技术需求文档
- **时间**: 1-2 天
- **检查点**:
  - [ ] 功能需求清晰
  - [ ] 非功能需求明确（性能、安全、可用性）
  - [ ] 约束条件识别（时间、成本、团队）

#### 1.2 架构设计
- **输入**: 技术需求文档
- **输出**: 架构设计文档
- **时间**: 3-5 天
- **检查点**:
  - [ ] 系统上下文图（C4 Level 1）
  - [ ] 容器图（C4 Level 2）
  - [ ] 组件图（C4 Level 3）
  - [ ] 数据流图
  - [ ] 部署架构图

#### 1.3 技术选型
- **输入**: 架构设计文档
- **输出**: 技术选型报告
- **时间**: 2-3 天
- **检查点**:
  - [ ] 候选方案对比矩阵
  - [ ] 推荐方案及理由
  - [ ] 风险评估
  - [ ] POC 验证计划（如需要）

#### 1.4 方案评审
- **输入**: 架构设计文档 + 技术选型报告
- **输出**: 评审意见、修订方案
- **时间**: 1 天
- **参与者**: 架构师、PM、技术 Leader、核心开发
- **检查点**:
  - [ ] 方案可行性确认
  - [ ] 风险可接受
  - [ ] 团队能力匹配
  - [ ] 成本在预算内

#### 1.5 文档输出
- **输出**:
  - 架构设计文档（最终版）
  - ADR 记录
  - 接口设计文档
  - 数据库设计文档
- **时间**: 1 天

#### 1.6 实施指导
- **活动**:
  - 向开发团队讲解架构
  - 解答技术疑问
  - 审查关键代码
- **持续时间**: 整个开发周期

## 2. 技术选型工作流

### 触发条件
- 引入新技术栈
- 替换现有技术
- 中间件选型
- 工具链选择

### 工作流程
```
选型需求 → 方案调研 → 对比评估 → POC 验证 → 决策记录 → 推广实施
```

### 详细步骤

#### 2.1 方案调研
- **时间**: 1-2 天
- **输出**: 候选方案列表
- **检查点**:
  - [ ] 至少 3 个候选方案
  - [ ] 每个方案的基本信息（功能、性能、生态）

#### 2.2 对比评估
- **时间**: 1-2 天
- **输出**: 评估矩阵
- **评估维度**:
  - 功能匹配度（25%）
  - 性能（20%）
  - 生态（15%）
  - 团队适配（15%）
  - 成本（10%）
  - 安全（10%）
  - 可扩展性（5%）

#### 2.3 POC 验证（可选）
- **时间**: 3-5 天
- **输出**: POC 报告
- **验证内容**:
  - 核心功能验证
  - 性能基准测试
  - 集成难度评估

#### 2.4 决策记录
- **输出**: ADR 文档
- **内容**:
  - 决策背景
  - 候选方案对比
  - 最终决策及理由
  - 预期影响

#### 2.5 推广实施
- **活动**:
  - 技术分享会
  - 编写使用文档
  - 提供技术支持

## 3. 代码审查工作流

### 触发条件
- Pull Request 提交
- 关键模块变更
- 架构相关修改

### 工作流程
```
PR 提交 → 自动检查 → 人工审查 → 反馈意见 → 修改确认 → 合并
```

### 审查清单

#### 3.1 架构一致性
- [ ] 符合架构设计
- [ ] 模块划分合理
- [ ] 接口设计规范
- [ ] 依赖关系清晰

#### 3.2 代码质量
- [ ] 符合代码规范
- [ ] 命名清晰
- [ ] 逻辑简洁
- [ ] 可维护性高

#### 3.3 安全性
- [ ] 无 SQL 注入
- [ ] 无 XSS 漏洞
- [ ] 输入验证完整
- [ ] 权限检查正确
- [ ] 敏感数据加密

#### 3.4 性能
- [ ] 无 N+1 查询
- [ ] 无内存泄漏
- [ ] 算法复杂度合理
- [ ] 数据库查询优化

#### 3.5 测试
- [ ] 单元测试覆盖 > 80%
- [ ] 关键路径有集成测试
- [ ] 测试用例充分

### 审查时间要求
- 普通 PR: 24 小时内
- 紧急 PR: 4 小时内
- 大型 PR (> 500 行): 48 小时内

## 4. 安全审查工作流

### 触发条件
- 上线前安全检查
- 安全漏洞修复
- 权限系统变更
- 敏感数据处理

### 工作流程
```
审查请求 → OWASP 检查 → 代码扫描 → 渗透测试 → 审查报告 → 修复验证
```

### 审查清单

#### 4.1 OWASP Top 10
- [ ] A01: 访问控制失效
- [ ] A02: 加密失败
- [ ] A03: 注入
- [ ] A04: 不安全设计
- [ ] A05: 安全配置错误
- [ ] A06: 过时组件
- [ ] A07: 认证失败
- [ ] A08: 数据完整性
- [ ] A09: 日志不足
- [ ] A10: SSRF

#### 4.2 代码扫描
- 使用工具: Bandit (Python) / ESLint Security (JS)
- 扫描范围: 所有代码
- 处理: 修复所有 Critical 和 High 级别问题

#### 4.3 审查报告
- **输出**: 安全审查报告
- **内容**:
  - 发现的问题清单
  - 风险等级评估
  - 修复建议
  - 修复优先级

## 5. 架构评审工作流

### 触发条件
- 架构设计完成
- 架构重大变更
- 定期架构审视

### 工作流程
```
评审准备 → 评审会议 → 意见收集 → 方案修订 → 最终确认
```

### 评审会议

#### 参与者
- 架构师（主持）
- 产品经理
- 技术 Leader
- 核心开发
- 测试负责人
- 运维负责人

#### 评审内容
1. 架构设计讲解（30 分钟）
2. 技术选型说明（15 分钟）
3. 风险评估（15 分钟）
4. 讨论和质疑（30 分钟）
5. 总结和决策（10 分钟）

#### 评审标准
- [ ] 满足业务需求
- [ ] 技术方案可行
- [ ] 团队能力匹配
- [ ] 风险可控
- [ ] 成本合理
- [ ] 时间可接受

## 6. 技术债务管理工作流

### 触发条件
- 定期技术债务审视（每季度）
- 性能问题积累
- 代码质量下降

### 工作流程
```
债务识别 → 优先级排序 → 制定计划 → 逐步偿还 → 效果评估
```

### 债务分类
- **架构债务**: 架构设计不合理
- **代码债务**: 代码质量差
- **测试债务**: 测试覆盖不足
- **文档债务**: 文档缺失或过时

### 偿还策略
- 高优先级: 立即处理
- 中优先级: 纳入迭代计划
- 低优先级: 择机处理

---

## 7. 故障复盘工作流

### 触发条件
- 生产环境故障
- 重大 Bug
- 性能问题

### 工作流程
```
故障发生 → 应急处理 → 根因分析 → 复盘会议 → 改进措施 → 跟踪落实
```

### 复盘会议

#### 参与者
- 架构师
- 相关开发
- 测试工程师
- 运维工程师

#### 复盘内容
1. 故障时间线
2. 根本原因分析
3. 影响范围评估
4. 应急处理回顾
5. 改进措施制定

#### 输出
- 故障复盘报告
- 改进措施清单
- 责任人和时间表
